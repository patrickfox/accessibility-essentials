Terminology

  • a11y = accessibility
  • AT = assistive technology
  • Screen reader = software that interprets code and reads content
  • WAI-ARIA = W3C technical spec, enables modern a11y
  • WCAG2 = W3C a11y guidelines, v2
  • Rehabilitation Act(1973), Section 508 = provides electronic a11y guidelines for US government vendors
  • Americans w/Disabilities Act (ADA, 1990) = law, prohibits discrimination

A11y is important

  • Companies: Legal, brand risks
  • People w/disabilities: Independence, dignity
  • A11y is an equality and civil rights issue
  • A11y benefits everyone:
    • temporary disabilities
    • curb cuts, door openers

ARIA Makes Modern A11y Possible

What makes accessible, modern web UI so challenging?

  • Dynamic UI: content updates, state changes not communicated
  • Custom UI: role and purpose of non-native widgets
  • ARIA helps bridge this gap
    • Live region: aria-live
    • States: aria-enabled, aria-selected
    • Roles: role="dialog"

Modern A11y Goals and Tips

  • Create lean, elegant, robust technical solutions
  • Minimize use of ARIA - rely on semantic code first
  • Aim for widest screen reader/browser support possible
  • Test often - don't wait until the end
  • Developers and QA peeps should be screen reader experts
  • Make accessibility easier, repeatable
  • Make a11y a core part of every step of project lifecycle
  • Accessibility is a team effort
  • Be wary of "accessible" frameworks - test for yourself, improve
  • Give back - share what you know, report issues

Tools and Techniques

Focus Management

Managing where focus is and avoiding the loss of focus

Tips:

  • Manage focus only to improve understanding, avoid frustration
  • Focusing on an element reads its contents or label
  • Only natively focusable elements or elements w/tabindex can receive focus

Using Tabindex

Tips:

  • Hard coding tabindex can be problematic
  • Do not use tabindex="0"
  • Use a script to dynamically manage focus…

    •

    Focus on anything w/ $.access()

    How it works:

    • Adds tabindex="-1"
    • Places focus on the target element
    • Listens for focusout event and cleans up everything

    Using $.access()

    Example Usage:

    • Focused item is no longer available
    • Modals
    • Single-page app view updates
    $("#home_section").access(); //→ places focus on the element $("#home_section").access(true); //→ places focus before the element $.access() in action →

    Announcing Content

    Live regions automatically speak content updates

    Use cases:

    • alerts
    • errors
    • notifications
    • any pertinent update

    Announcing Content

    Implementation:

    # aria-live={polite || assertive}
    ...
    # log is shorthand for aria-live="polite"
    ...
    # alert is shorthand for aria-live="assertive"
    # status is shorthand for aria-live="polite"

    One Live Region To Rule Them All

    Live region pitfalls:

    • Multiple live regions cumbersome to manage
    • Live region roles(status, alert, log, timer) don't work reliably
    • Ad-hoc live region use adds complexity, risk

    Best practices:

    • Use only ONE live region in your entire site/app
    • Use a script to announce content…

    Say Anything w/$.announce()

    How it works:

    • Create common @aria-live “announcer” element
    • Announcer needs to be in DOM on page load and left alone
    • Reset technique allows repeat announcements

    Isolate and Tab Fence

    Temporarily isolate user action/focus to specified area of UI

    • Isolation uses aria-hidden to hide page content
    • Tab Fence listens for focus on page, returns focus to isolated UI

    Usage:

    • Modals
    • In-page, multi-step process

    Alternate Copy and Labeling

    Additional non-visual copy used to improve accessibility

    Uses:

    • Summarizing complex copy
    • Translating visual cues to spoken cues
    • Providing additional content to improve meaning

    Tips:

    • Labeled elements are usually interactive
    • Minimize use of alternate content

    Alternate Copy and Labeling

    Techniques:

    • aria-label="alternate copy" - works well on buttons
    • aria-labelledby="id/id's" - associate existing content to an element
    • aria-describedby="id/id's" - additional descriptive content
    • .visuallyhidden CSS technique - can be problematic on iOS
    • title - not reliable, don't use!

    Alternate Copy and Labeling

    Implementation:

    #label an element using another element's copy <div aria-labelledby="some_id another_id third_id"> #additional descriptive content <input id="first_name" aria-describedby="visual_descriptive_text" type="text" /> aria-labelledby example →

    Responsive Design Considerations

    Common codebase === more testing, more complexity

    • Avoid overly complex UI changes - keep content flow the same
    • Touch gestures require accessible equivalent
    • Manage state and functionality changes w/JS - Hamburger Menu →
    • Minimize swipes on mobile - simplify and use semantic markup
    • aria-hidden="false" breaks iOS VO
    • Test each break point…

    CSS Pseudo Elements

    Content via CSS (:before, :after)

    Tips:

    • Screen readers support and read CSS content
    • Do not use with pertinent content
    • Great for icon fonts, but be careful...
      • Icon fonts rely on Unicode character mapping
      • MacOS VO used to read some Unicode characters as an audible beep
      • Some Unicode characters map to emoji →
      • Use a service like Icomoon

    Implementing A11y

    "Semantic" Markup

    Semantic markup is the foundation of a11y, but...

    • Generic tags are sometimes better, less verbose
    • Most important: How is markup spoken?
    • Test options, use what sounds best
    • Examples:
      • HTML5 sectioning elements spoken as landmarks
      • Use buttons & anchors vs. generic elements
      • UL/LI for true list content ONLY
      • DL/DD/DT are overly verbose

    Headings

    Headings are crucial to page navigation and content hierarchy

    • UX team should determine heading strategy
    • Headings describe sections or groups of content
    • Order should not increase by more than 1
    • Don't overuse
    • Don't put links/buttons in headings

    Landmarks

    Landmarks improve navigation and understanding of page structure

    Role HTML Tag Usage Notes
    banner <header role="banner"> once per view Page header
    main <main role="main"> once per view Main content area
    contentinfo <footer role="contentinfo"> once per view Page footer
    search - multiple Search form
    form - multiple Don't use unless a form plays a significant role in page
    navigation <nav role="navigation"> multiple Use with label/heading
    complementary <aside role="complementary"> multiple Use with label/heading
    application - !avoid

    Landmark Usage

    At a minimum, all pages should have:

    • banner - <header role="banner">
    • main - <main role="main">
    • contentinfo - <footer role="contentinfo">

    Goal: ensure that all content resides in a landmark region and, in doing so, no content is orphaned

    Landmark Tips

    • Create a landmark strategy, use it consistently
    • Use labels for generic landmarks (e.g. navigation, complementary)
    • Some HTML5 sectioning elements are landmarks by default
    • Use HTML5 element and role
    • Don't overuse

    Modals

    Creating accessible modals

    Modal setup:

    • start heading order with H1
    • use aria-labelledby to label modal
    • add role=dialog to container

    On open:

    • focus on first focusable or dialog body
    • isolate modal UI from view content
    • dialog is announced

    On close:

    • return focus to button, last focused or main content

    Deep Dive: Responsive Menus

    Starbucks.com

    Overall, site is pretty solid

    • Responsive
    • Accessible
    • A couple of things though...
    Starbucks.com →

    Starbucks.com

    Problem: hard coded tabindex causes issues

    • 6 tabindex="0" on main content areas
    • 4 tabindex="-1" on other areas

    Solution: use $.access() to manage focus!

    Starbucks.com

    Problem: Hidden menu items still navigable

    • CSS transitions used to animate menu open/close
    • aria-hidden is problematic in VoiceOver
    • display:none; should be used to hide menus when closed
    • Timing is key: hide menus after CSS transition ends

    Starbucks.com

    Solution: Listen for transitionend/webkitTransitionEnd event on menu close

    • Event fires on end of CSS transition
    • Set menu display display:none;
    • Note: removing CSS transitions will break menu toggle...

    • Announce and access your modern UI w/a11y_kit
      • Don't hard code tabindex - use $.access() to manage focus

    • Use aria-labelledby to summarize complex content
    • Keep your code lean
    • Minimize use of ARIA and alternate copy
    • Maximize foundation of semantic markup, and solid design and content strategy
