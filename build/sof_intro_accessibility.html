Introduction

Terminology

  • a11y = accessibility
  • AT = assistive technology
  • Screen reader = software that interprets code and reads content
  • WAI-ARIA = W3C technical spec, enables modern a11y
  • WCAG2 = W3C a11y guidelines, v2
  • Rehabilitation Act(1973), Section 508 = provides electronic a11y guidelines for US government vendors
  • Americans w/Disabilities Act (ADA, 1990) = law, prohibits discrimination

What is accessibility?

Accessibility is important

  • for our clients: Legal, brand risks
  • for Razorfish: New business, value-add, trust
  • A11y benefits everyone:
    • independence, dignity
    • temporary disabilities
    • curb cuts, door openers
  • A11y is an equality and civil rights issue
"When we work on making our devices accessible by the blind, I don't consider the bloody ROI".
- Tim Cook, Apple CEO

Accessibility is inclusivity

The goal of web accessibility is to include people of all abilities:

  • Visual Color blindness, partial or total blindness
  • Audible Sound communicated via text
  • Mobility Limited or loss of limb use
  • Cognitive ADD/ADHD, dyslexia, learning disabilities

Accessibility requires awareness and empathy

  • Compliance vs. Usability Not enough to be compliant
  • Team effort: Designers, IA's, copywriters, and developers
  • Ensure that each componenet and aspect of your site/app is:
    • perceivable
    • operable
    • understandable
    • robust

    Modern A11y Goals and Tips

    • Create lean, elegant, robust technical solutions
    • Minimize use of ARIA - rely on semantic code first
    • Aim for widest screen reader/browser support possible
    • Test often - don't wait until the end
    • Developers and QA peeps should be screen reader experts
    • Make a11y a core part of every step of project lifecycle
    • Accessibility is a team effort
    • Make accessibility easier, repeatable
    • Be wary of "accessible" frameworks - test for yourself, improve
    • Give back - share what you know, report issues

    Modern Web UI Challenges

    Announcing Content

    Live regions automatically speak content updates

    Use cases:

    • alerts
    • errors
    • notifications
    • any pertinent update

    Announcing Content

    Implementation:

    # aria-live={polite | assertive | off}
    ...
    # log is shorthand for aria-live="polite"
    ...
    # alert is shorthand for aria-live="assertive"
    # status is shorthand for aria-live="polite"

    One Live Region To Rule Them All

    Live region pitfalls:

    • Multiple live regions cumbersome to manage
    • Ad-hoc live region use adds complexity, risk
    • Live region roles(status, alert, log, timer) don't work reliably

    Best practices:

    • Use only ONE live region in your entire site/app
    • Use a common script to announce content…

    Say Anything w/$.announce()

    How it works:

    • Create common @aria-live “announcer” element
    • Announcer needs to be in DOM on page load and left alone
    • Reset technique allows repeat announcements

    Single Page App A11y

    Single Page Architecture

    Intial page load, subsequent dynamic view updates via Ajax

    Problem: No page refresh - screen reader unaware of view updates

    Solution: Emulate native refresh, $.announce() status updates on view loaded

    • Update page title from view to view
    • $.announce() "{new title} view loaded" on "view loaded/updated" event
    • Use $.access() to dynamically place focus on new content area
    Screen reader page load behaviors →

    Example: AngularJS

    • Update and announce title on $viewContentLoaded
    app.controller "PageCtrl", ($scope, $location, $http) -> $scope.$on '$viewContentLoaded', announce_view_loaded ... Accessible Angular App →

    Example: React

    • Update and announce title via componentDidUpdate
    componentDidUpdate: -> //NOTE: make sure this is only called for *view* updates @make_announcement() @setPageScroll() make_announcement: (message) -> message = message || @getViewUpdateMsg() $.announce(message, 'assertive')

    Summary

    Cheat Sheet

    • Minimize use of ARIA
    • Maximize foundation of semantic markup, and solid design and content strategy
    • Announce and access your modern UI w/a11y_kit, judiciously…
      • One live region for your app - use $.announce()
      • Don't hard code tabindex - use $.access() to manage focus
    • Keep your code lean and maintainable
