Introduction

Terminology

  • a11y = accessibility
  • AT = assistive technology
  • Screen reader = software that interprets code and reads content
  • WAI-ARIA = W3C technical spec, enables modern a11y
  • WCAG2 = W3C a11y guidelines, v2
  • Rehabilitation Act(1973), Section 508 = provides electronic a11y guidelines for US government vendors
  • Americans w/Disabilities Act (ADA, 1990) = law, prohibits discrimination

What is accessibility?

Accessibility is inclusivity

Account for wider range of abilities:

  • Visual Color blindness, partial or total blindness
  • Aural Sound communicated via text
  • Physical Limited or loss of limb use
  • Cognitive ADD/ADHD, dyslexia, learning disabilities

Accessibility is important

  • for our clients: Legal, brand risks
  • for Razorfish: New business, value-add, trust
  • A11y benefits everyone:
    • independence, dignity
    • temporary disabilities
    • curb cuts, door openers
  • A11y is an equality and civil rights issue
"When we work on making our devices accessible by the blind, I don't consider the bloody ROI".
- Tim Cook, Apple CEO

Accessibility requires awareness and empathy

  • Compliance vs. Usability Not enough to be compliant
  • Team effort: Designers, IA's, copywriters, and developers
  • Ensure that the structure and each component of your site/app is:
    • perceivable
    • operable
    • understandable
    • robust

Accessibility Goals - POUR

Users of varying abilities can percieve, operate and understand your UI

Solutions are robust and can work across a spectrum of screen readers and browsers

Accessibility Goals

  • Ensure device independence
    • site functionality operable via keyboard or mouse
  • Provide meaningful, logical content structure and order
    • semantic markup
    • navigable and discoverable UI
    • content flow matches visual flow
    • screen readers can speak your UI
  • Provide equivalent content/experiences
    • visual cues/associations translate to spoken experience
    • color contrast
    • alternate text

Accessibility Standards

W3C WCAG2

  • W3C, international standard
  • Levels of compliance: A, AA and AAA

WCAG2 - Percievable

1.1 Non-text Content

  • Add @alt to all image tags - alt="" is legit
  • Ensure that all form inputs have text labels

WCAG2 - Percievable

1.2 Time-based Media: Provide alternatives for time-based media

  • Provide captions and transcripts for video

WCAG2 - Percievable

1.3 Adaptable: Create content that can be presented in different but equivalent ways

  • Create a semantic structure that represents the purpose of the content*
  • Create a logical heading hierarchy and tab order
  • Use tables for tabular data only
  • Associate text labels with form inputs
  • Ensure logical and intuitive reading and navigation order(determined by code order)
  • Ensure that instructions do not rely upon shape, size, color or visual location

WCAG2 - Percievable

1.4 Distinguishable: Make it easier for users to see and hear content including separating foreground from background

  • Provide text alternatives for color-based indicators
  • Ensure accessible color contrast:
    • Small text(< 18pt): 4.5:1 contrast ratio
    • Large text (>= 18pt): 3:1 contrast ratio

WCAG2 - Operable

2.1 Keyboard Accessible: Make all functionality available from a keyboard

  • Ensure keyboard access
  • Emulate native keyboard navigation
  • Use ARIA roles to define widgets, interactive elements

WCAG2 - Operable

2.2 Enough Time: Provide users enough time to read and use content

  • Communicate and allow extension of time limits
  • Allow user to control animating content(e.g. carousels)

WCAG2 - Operable

2.3 Seizures: Do not design content in a way that is known to cause seizures

  • Avoid flashing content(<=3 Hz)

WCAG2 - Operable

2.4 Navigable: Provide ways to help users navigate, find content, and determine where they are

  • Provide skip navigation
  • Provide descriptive and informative page titles, headings, form labels, other controls
  • Provide logical and intuitive focus/tab and heading order*
  • Ensure that link/button(anything tabable) text stands on it own
  • Ensure visual focus state
  • Use ARIA landmark roles to identify various sections of the page

WCAG2 - Understandable

3.1 Readable: Make text content readable and understandable

  • Page language is identified
  • Provide accessible instructions and contextual info via .visuallyhidden content

WCAG2 - Understandable

3.2 Predictable: Make Web pages appear and operate in predictable ways

  • Avoid changing context upon user focus or input
  • Communicate significant status or content updates
  • Reduce audible clutter by providing concise, non-repetitive content
  • Provide consistent navigation

WCAG2 - Understandable

3.3 Input Assistance: Help users avoid and correct mistakes

  • Provide accessible input requirements and form validation
  • Provide suggestions/help for errors

Section 508

  • US standard
  • Changing soon to adopt WCAG2 standard
  • Roughly maps to WCAG2 Level A

Compliance vs. Usability

  • Inclusivity and usability vs. specific wording of guidelines

WAI ARIA

Modern A11y Challenges

Dynamic UI: content updates, state changes not communicated

  • Dynamic errors/notifications
  • View updates in single-page apps

Custom UI: role and purpose of non-native widgets not understood

  • Custom drop-down menus
  • Modals
  • Tooltips
  • Accordion content

ARIA Powers Modern A11y

ARIA specifies a set of attributes that allow us to set and communicate:

  • States: aria-enabled, aria-selected
  • Properties: aria-haspopup
  • Roles and Landmarks: role="dialog", role="contentinfo"
  • Relationships: aria-labelledby, aria-controls
  • Live Regions: aria-live

Screen Readers and Browsers

Page Navigation

Two types of focus:

Browser focus
  • Natively focusable elements(input, links, buttons, etc.)
  • Use tab key to navigate
Screen reader(virtual focus)
  • Navigate element-by-element
  • Specific elements
    • Headings
    • Landmarks
    • Form elements
    • Links, buttons
  • Use arrow keys and modifiers to explore structure

Screen Reader Stats

Desktop

  • JAWS(Win) ~50%, mostly in US
  • NVDA(Win) ~20%, higher in EU
  • VoiceOver(MacOS) ~10%

Mobile

  • VoiceOver(iOS) ~60%, higher in US
  • Talkback(Android) ~20%

Source: WebAIM.org Screen Reader Survey 5

Screen Reader Testing

  • Test early and often!
  • Support for a11y features varies
  • Recommended Browser/Screen Reader Combos:
    • NVDA / Firefox (Windows)
    • JAWS / IE (Windows)
    • VoiceOver / Safari (MacOS/iOS)
    • TalkBack / Firefox (Android)

Implementation - Core Accessibility

Headings

Importance

  • Headings and heading order are important to understanding page hierarchy
  • AT allows users to navigate pages via headings

Usage

  • Use headings consistently from page to page
  • Heading order must not increase by more than one
  • Good: h1, h2, h2, h3, h4, h2, h3
  • Not good: h1, h2, h4, h4, h2, h5
  • One H1 per page*(per Citi team)
  • Modals should always start w/H1

Handle with care

  • Add headings to your UI only if they are part of a larger strategy
  • Don't add headings to the page randomly - follow the copy deck

Landmarks

Landmarks improve navigation and understanding of page structure

Role HTML Tag Usage Notes
banner <header role="banner"> once per view Page header
main <main role="main"> once per view Main content area
contentinfo <footer role="contentinfo"> once per view Page footer
search - multiple Search form
form - multiple Don't use unless a form plays a significant role in page
navigation <nav role="navigation"> multiple Use with label/heading
complementary <aside role="complementary"> multiple Use with label/heading
application - !avoid

Landmark Usage

At a minimum, all pages should have:

  • banner - <header role="banner">
  • main - <main role="main">
  • contentinfo - <footer role="contentinfo">

Goal: ensure that all content resides in a landmark region and, in doing so, no content is orphaned

Landmark Tips

  • Create a landmark strategy, use it consistently
  • Use labels for generic landmarks (e.g. navigation, complementary)
  • Some HTML5 sectioning elements are landmarks by default
  • Use HTML5 element and role
  • Don't overuse

Tables

Table Headings

  • TD's must be associated with any row or column headings via proper use of TH.
  • TH's are used by screen readers to label each TD
  • Use the scope attribute to denote that a header applies to the entire row or column

  • Use table headings(<th>) to label rows and columns

Forms

Labeling

All form inputs must have a label(e.g. <label>)

<div> <label for="url">Website: </label> <input type="url" name="url" id="url"> </div> <div> <input aria-label="Zip Code" type="text" name="zip" id="zip"> </div>

Grouping

Associated form controls should be grouped using <fieldset> and <legend>

<fieldset> <legend>Output format</legend> <div> <input type="radio" name="format" id="txt" value="txt" checked> <label for="txt">Text file</label> </div> <div> <input type="radio" name="format" id="csv" value="csv"> <label for="csv">CSV file</label> </div> </fieldset>

Alternate Text

Use Cases

  • Translating visual content to spoken content
  • Provide additional information for interactive elements:
    • links
    • buttons
    • form elements

Images

  • <img> always requires alt attribute
  • Alt text speaks for image content
  • Non-essential images should have alt=""

Labeling Techniques

  • aria-label="{label text}" -
  • aria-labelledby="{id list}" - use other element(s) as label
  • aria-describedby="{id list}" - use other element(s) as description
  • .visuallyhidden - CSS technique, visually hides content, available to screen reader

Focus Management

What is focus management?

Care and management of where focus is

Placing focus on an element:

  • speaks the contents of that element
  • reads associated label text - aria-label, aria-labelledby
  • moves keyboard focus where we want it to be

Tips:

  • don't move keyboard focus unless necessary
  • take care not to lose keyboard focus
  • placing focus can help communicate context changes

Modals

Modal Goals

  • Focus and maintain user's attention on modal UI
  • Prevent interaction with background content
    • Tabbing
    • Screen reader virtual navigation

Modal Setup

<div role="dialog" aria-labelledby="label" aria-describedby="description"> <h1 id="label">Account Alert: Account Overdrawn</h1> <p id="description">Your account ending in 4567 has a negative balance of -$15.76</p> <button>Transfer Funds</button> <button>Remind Me Later</button> </div>
  • Start heading order with H1
  • role="dialog" announces "dialog"
  • aria-labelledby="label" announces #label
  • aria-describedby="description" announces #description

End result: "Account Overdrawn dialog Your account ending in 4567 has a negative balance of -$15.76, Transfer Funds button"

Modal Open

  • Focus on first focusable or dialog body
  • Isolate modal UI from view content(aria-hidden)
  • Dialog is announced

Modal Close

  • Return focus to initiating button, last focused or main content
  • Reset everything

Modal Isolation Detail

#all main page content lives here
#modals live at body end, hidden until activated
...
  • Main content lives in one element - div(data-isolate)
  • Hide page content from screen reader - aria-hidden=true
  • Focus listener returns page focus to modal

Modern A11y Goals and Tips

  • Create lean, elegant, robust technical solutions
  • Minimize use of ARIA - rely on semantic code first
  • Aim for widest screen reader/browser support possible
  • Test often - don't wait until the end
  • Developers and QA peeps should be screen reader experts
  • Make a11y a core part of every step of project lifecycle
  • Accessibility is a team effort
  • Make accessibility easier, repeatable
  • Be wary of "accessible" frameworks - test for yourself, improve
  • Give back - share what you know, report issues
