Introduction

So, what is this workshop about?

  • Modern accessibility challenges and solutions
  • Technical considerations and best practices

Terminology

  • a11y = accessibility
  • AT = assistive technology
  • Screen reader = software that interprets code and reads content
  • WAI-ARIA = W3C technical spec, enables modern a11y
  • WCAG2 = W3C a11y guidelines, v2

Goals of A11y

  • Inclusivity, equality
  • Exceed the standards - focus on empathy, usability
  • Ensure that screen readers, browsers and code are able to communicate UI structure, states, properties to users
  • Device independent navigation

"Accessibility is usability in context of disability"

"Empathy encourages you to understand who uses your product, forces you ask deeper questions and motivates the many redesigns you go through to get a product right."

- Scott Jensen, bit.ly/a11y-empathy

Modern A11y Challenges

Dynamic UI: content updates, state changes not communicated

  • Dynamic errors/notifications
  • View updates in single-page apps

Custom UI: role and purpose of non-native widgets not understood

  • Custom drop-down menus
  • Modals
  • Tooltips
  • Accordion content

ARIA Powers Modern A11y

ARIA specifies a set of attributes that allow us to communicate:

  • States: aria-enabled, aria-selected
  • Properties: aria-haspopup
  • Roles: role="dialog"
  • Relationships: aria-labelledby, aria-controls
  • Live region: aria-live

Screen Readers

  • Speak document content
  • Provide navigation tools
    • Headings
    • Landmarks
    • Forms and form fields
    • Links, buttons
    • Element by element
  • A11y feature support varies

Modern A11y Goals and Tips

  • Create lean, elegant, robust technical solutions
  • Minimize use of ARIA - rely on semantic code first
  • Aim for widest screen reader/browser support possible
  • Test often - don't wait until the end
  • Developers and QA peeps should be screen reader experts
  • Make accessibility easier, repeatable
  • Make a11y a core part of every step of project lifecycle
  • Accessibility is a team effort
  • Be wary of "accessible" frameworks - test for yourself, improve
  • Give back - share what you know, report issues

Modern A11y Techniques

Announcing Content

Live regions automatically speak content updates

Use cases:

  • alerts
  • errors
  • notifications
  • any pertinent update

Announcing Content

Implementation:

# aria-live={polite || assertive}
...
# log is shorthand for aria-live="polite"
...
# alert is shorthand for aria-live="assertive"
# status is shorthand for aria-live="polite"

One Live Region To Rule Them All

Live region pitfalls:

  • Multiple live regions cumbersome to manage
  • Live region roles(status, alert, log, timer) don't work reliably
  • Ad-hoc live region use adds complexity, risk

Best practices:

  • Use only ONE live region in your entire site/app
  • Use a script to announce content…

Say Anything w/$.announce()

How it works:

  • Create common @aria-live “announcer” element
  • Announcer needs to be in DOM on page load and left alone
  • Reset technique allows repeat announcements

Single Page Architecture

Intial page load, subsequent dynamic view updates via Ajax

Problem: No page refresh - screen reader unaware of view updates

Solution: Emulate native refresh, $.announce() status updates on view loaded

  • Update page title from view to view
  • $.announce() "{new title} view loaded" on "view loaded/updated" event
  • Use $.access() to dynamically place focus on new content area

Accessible Modals

Example usage:
  • user or system initiated prompts
  • display of extra details

    •

    Modal Goals

    • Prevent confusion
    • Focus and maintain user's attention on modal UI
    • Prevent interaction with background content
      • Tabbing
      • Screen reader virtual navigation
    • Commnicate modal presence and purpose

    Modal Setup

    • start heading order with H1
    • use aria-labelledby to label modal
    • add role=dialog to container

    Modal Open

    • focus on first focusable or dialog body
    • isolate modal UI from view content(aria-hidden)
    • dialog is announced

    Modal Close

    • return focus to button, last focused or main content
    • reset everything

    Modal Isolation Detail

    #all main page content lives here
    #modals live at body end, hidden until activated
    ...
    • Main content lives in one element - div(data-isolate)
    • Hide page content from screen reader - aria-hidden=true
    • Focus listener returns page focus to modal
